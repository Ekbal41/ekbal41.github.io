Introduction

React makes building user interfaces a breeze—but it’s also easy to write code that looks fine but performs poorly under the hood. Whether you’re building a large-scale app or a side project, performance issues can creep in from innocent mistakes. In this post, we’ll go through 10 common React performance pitfalls, why they matter, and how to fix them. Let’s dive in.

1. Unnecessary Re-renders from Inline Functions and Objects

The Problem

When you define functions or objects directly inside a component’s render (or return) block, React treats them as new references on every render. This causes child components that depend on those props to re-render unnecessarily—even if nothing else has changed.

2. Overusing Context API for High-Frequency Updates

The Problem

React’s Context API is great for avoiding prop drilling, but it’s not always the best tool for frequently changing values—like live input, theme toggles, or user typing. When context values change, every component that consumes the context will re-render, even if only one small part of the tree needed the update.

This can lead to performance bottlenecks, especially in large apps.

If count changes rapidly, the entire DeepTree subtree re-renders every time.

The Fix

Use context for rarely changing state, like themes or auth. For dynamic values, consider:

  • Using useReducer or useState with prop drilling (it’s not always bad)
  • Lifting state only as far as necessary
  • Using libraries like Zustand or Jotai for local/global state management with minimal re-renders

Keep your contexts lean and avoid using them for high-frequency state.

3. Skipping Memoization for Expensive Computations

The Problem

In React, expensive computations run on every render unless they’re memoized. This can include things like filtering large arrays, calculating derived data, or formatting complex outputs. If left unchecked, this slows down rendering performance significantly.

Every render re-filters the list, even if items hasn’t changed.

The Fix

Use useMemo to cache the result of expensive computations unless dependencies change.

This ensures the filtering only runs when items actually changes, reducing unnecessary CPU usage.

4. Not Using React.memo for Pure Functional Components

The Problem

By default, functional components in React re-render whenever their parent re-renders, even if their props haven’t changed. For pure components—ones that render the same output for the same props—this is wasteful.

Even if user is the same, UserCard re-renders with its parent.

The Fix

Wrap pure functional components with React.memo to memoize them. This tells React to skip re-rendering if props haven’t changed.

Make sure props are stable and comparable (i.e., primitive values or memoized objects) for React.memo to be effective.

5. Forgetting to Use Key Props Correctly in Lists

The Problem

React uses the key prop to track which items have changed, been added, or removed in a list. If you forget to add keys—or use unstable values like array indexes—React can’t reliably track elements, leading to unnecessary re-renders or even bugs in UI behavior.

Using index works fine for static lists but causes problems when the list is reordered, filtered, or dynamically updated.

The Fix

Always use a unique, stable identifier for the key prop.

This helps React optimize rendering and ensures consistent behavior, especially during state updates or animations.

6. Excessive Use of useEffect for Side Effects

The Problem

useEffect is a powerful hook in React for handling side effects, like fetching data, subscribing to events, or manually manipulating the DOM. However, overusing useEffect or using it for non-essential tasks can hurt performance, especially if it’s called on every render.

Calling useEffect on every render with empty dependencies (or without proper dependency checks) can result in redundant operations like unnecessary API calls or state updates.

The Fix

Only use useEffect for tasks that truly require side effects. If you don’t need to trigger an effect on every render, specify dependencies to ensure it only runs when necessary.

For side effects related to state changes, consider using event listeners or memoized callbacks outside of useEffect to avoid unnecessary re-renders.

7. Not Leveraging Code Splitting for Large React Apps

The Problem

React applications can become quite large, especially as you add more features and dependencies. If you don’t leverage code splitting, your users will have to download the entire app bundle upfront, even if they only need a small portion of it. This increases the initial load time and decreases the perceived performance of your application.

If your app grows, this could lead to long loading times as your app package grows in size.

The Fix

Leverage dynamic import() to split your code into smaller chunks, loading them only when needed. You can use React.lazy and Suspense to easily implement this in React.

This approach allows React to load components only when they are needed, resulting in smaller initial bundles and faster loading times for your users.

8. Ignoring Accessibility in React Applications

The Problem

Accessibility (a11y) is often overlooked in React applications, leading to poor user experiences for individuals with disabilities. This can include missing alt text for images, improper use of ARIA roles, or ignoring keyboard navigation. Accessibility issues not only hurt users but can also affect your app’s reach and SEO rankings.

This is problematic for users with visual impairments who rely on screen readers.

The Fix

Make sure your app is accessible by following best practices for accessibility:

  • Provide meaningful alt text for all images.

  • Use semantic HTML elements like <header>, <nav>, and <footer> to improve structure.

  • Ensure that interactive elements are keyboard accessible (e.g., using tabIndex).

Additionally, use tools like React Axe to identify accessibility issues during development.

By prioritizing accessibility, you’ll create a better experience for all users and increase the inclusivity of your app.

9. Overcomplicating Component Structure

The Problem

React encourages building small, reusable components, but it’s easy to take this too far by overcomplicating your component structure. Having too many small components or deeply nested components can make your app harder to maintain and more difficult to understand. It also leads to unnecessary re-renders if the state is not properly managed.

In cases where you over-nest components, it’s easy to lose track of data flow and component responsibilities, leading to a confusing and hard-to-manage codebase.

The Fix

Aim for a balance between small, reusable components and maintainability. When creating components, make sure they are responsible for a clear and simple task. Avoid deeply nesting components unless necessary.

Instead of splitting components too finely, consider the following strategies:

  • Combine related logic and presentation in the same component.

  • Use custom hooks for shared logic.

  • Keep your component hierarchy shallow and logical.

By keeping your component structure simple and organized, you’ll avoid unnecessary complexity and improve the maintainability of your app.

10. Using Index as a Key in Lists

The Problem

When rendering lists of components in React, you may be tempted to use the index of the array as the key prop. While this might work for small, static lists, it causes problems when the list changes (e.g., items are added, removed, or reordered). React uses the key to identify components and optimize re-renders. If you use the index, React may not correctly track the components, leading to unexpected behavior, performance issues, and bugs.

In cases where items are reordered or removed, React might not correctly preserve the state of each item, leading to UI glitches or performance degradation.

The Fix

Always try to use a unique, stable identifier as the key prop, such as an ID from your data. This ensures that React can track each component correctly and optimize re-renders.

By using a stable key, you ensure that React can efficiently update only the components that need to be re-rendered.

Conclusion

In React development, performance is crucial to delivering a smooth and responsive user experience. By avoiding common mistakes, like overusing the Context API, neglecting memoization, or overcomplicating your component structure, you can significantly improve the performance of your application. Always be mindful of best practices, such as using React.memo for pure components, leveraging dynamic imports for lazy loading, and ensuring accessibility. These optimizations not only boost performance but also enhance the overall user experience.
